# Meegle vs Lark Base 比較調査レポート
## 建設業における工程管理・作業計画の最適ソリューション検討

**調査日**: 2025年12月21日
**目的**: 建設業の工程管理・作業計画において、Meegle と Lark Base のどちらが最適かを評価

---

## 1. エグゼクティブサマリー

### 結論

| 用途 | 推奨ツール | 理由 |
|-----|----------|------|
| **マスタデータ管理** | Lark Base | データベース機能、リンク・Lookup |
| **複雑な工程管理** | Meegle | ノード駆動型ワークフロー |
| **シンプルな進捗管理** | Lark Base | 導入容易、低コスト |
| **大規模プロジェクト** | Meegle | リソース管理、依存関係可視化 |
| **資格・力量管理** | Lark Base | 日付トリガー自動化 |

### 推奨アプローチ

**ハイブリッド運用**を推奨:
- **Lark Base**: マスタデータ（従業員・資格・案件）管理
- **Meegle**: 工程管理・タスク進捗管理

## 2. Meegle（ミーグル）概要

### 2.1 製品概要

| 項目 | 内容 |
|-----|------|
| 提供元 | Lark Japan株式会社 |
| カテゴリ | プロジェクト管理プラットフォーム |
| 特徴 | ノード駆動型ワークフローエンジン |
| 料金 | 20名まで無料、有料プランあり |

### 2.2 主要機能

#### ワークフロー管理
```
┌─────────────────────────────────────────────────────────────┐
│                  Meegle ワークフロー概念図                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [計画] ──→ [設計] ──→ [施工] ──→ [検査] ──→ [完了]       │
│     │          │          │          │                     │
│     ▼          ▼          ▼          ▼                     │
│   担当A      担当B      担当C      担当D                    │
│                                                             │
│   ※ 各ノードに担当・期限・依存関係を設定                    │
└─────────────────────────────────────────────────────────────┘
```

- **ノード駆動型**: プロジェクトを「ノード」と「フロー」で可視化
- **依存関係管理**: 工程間の前後関係を明確化
- **ボトルネック検出**: 遅延箇所を自動特定

#### ビュー機能

| ビュー | 用途 | 建設業での活用 |
|-------|------|--------------|
| ガントチャート | 全体スケジュール | 工程表として使用 |
| カンバン | タスク進捗 | 日次作業管理 |
| テーブル | 一覧表示 | レポート作成 |
| ツリー | 階層構造 | WBS管理 |

#### 自動化機能

```
トリガー例:
- タスクのステータスが「完了」になったら
  → 次工程の担当者に自動通知
  → 次タスクを自動開始

- 期限の2日前になったら
  → 担当者にリマインダー送信

- 進捗率が50%を超えたら
  → 管理者に報告
```

### 2.3 Meegleの強み

1. **複雑なプロジェクト管理に最適**
   - 大規模・長期プロジェクト向け
   - 複数チーム・部門の協働

2. **リソース管理**
   - メンバーの稼働状況を可視化
   - 負荷分散の最適化

3. **リスク管理**
   - 遅延の早期検出
   - 依存関係の問題を自動警告

4. **標準機能が充実**
   - プラグイン追加不要
   - ガントチャート・レポート標準搭載

### 2.4 Meegleの弱み

1. **学習コスト**
   - ワークフロー設計に習熟が必要
   - 初期セットアップに時間がかかる

2. **マスタデータ管理**
   - データベース機能はLark Baseに劣る
   - リンク・Lookup機能が限定的

3. **建設業向けテンプレート**
   - ソフトウェア開発向けが中心
   - 建設業向けカスタマイズが必要

## 3. Lark Base 概要

### 3.1 製品概要

| 項目 | 内容 |
|-----|------|
| 提供元 | Lark Japan株式会社 |
| カテゴリ | ノーコードデータベース |
| 特徴 | 表計算 + データベースのハイブリッド |
| 料金 | Lark基本機能に含まれる（無料枠あり） |

### 3.2 主要機能

#### データベース機能

```
┌─────────────────────────────────────────────────────────────┐
│                  Lark Base テーブル関係図                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [従業員マスタ] ←──リンク──→ [資格記録]                     │
│        │                         │                         │
│        ▼                         ▼                         │
│   [案件管理] ←────リンク────→ [工程管理]                     │
│                                                             │
│   ※ Lookup で関連データを自動参照                           │
└─────────────────────────────────────────────────────────────┘
```

- **リンクフィールド**: テーブル間の関連付け
- **Lookup**: 関連テーブルから値を自動取得
- **数式フィールド**: 自動計算

#### ビュー機能

| ビュー | 用途 |
|-------|------|
| グリッド | 表形式での編集 |
| カンバン | ステータス別管理 |
| カレンダー | 日程管理 |
| ガント | 工程表（基本機能） |
| ギャラリー | カード形式表示 |
| フォーム | データ入力用 |

#### オートメーション

```
トリガータイプ:
- 新しいレコードが追加されるとき
- レコードの内容が変更されるとき
- レコードが指定時刻になったとき（★重要）
- 固定時刻になったとき（毎日/毎週）
- ボタンがクリックされたとき

アクション:
- Larkメッセージを送信
- メールを送信
- レコードを追加/編集
- Webhook送信
```

### 3.3 Lark Baseの強み

1. **マスタデータ管理に最適**
   - 従業員・資格・顧客などの管理
   - リンク・Lookupで2重入力削減

2. **導入が容易**
   - Excelライクな操作性
   - 学習コストが低い

3. **日付トリガーが優秀**
   - 資格期限30日前に通知
   - 工程予定日にアラート

4. **Lark統合**
   - チャット・会議・ドキュメントと連携
   - 通知がLarkに直接届く

5. **コストパフォーマンス**
   - Lark基本機能に含まれる
   - 追加費用なし

### 3.4 Lark Baseの弱み

1. **複雑なワークフロー管理**
   - 工程間の依存関係表現が限定的
   - ノード駆動型ワークフローなし

2. **大規模プロジェクト**
   - リソース管理機能が限定的
   - 複数プロジェクト横断が困難

3. **ガントチャート機能**
   - 基本機能のみ
   - Meegleほど高機能ではない

```

│  Lark Base  │ ←→ │   Meegle    │

## 7. 現状の「シンプル建設業務管理パッケージ v2.0」の評価

### 7.1 現在の構成（Lark Base）

```
テーブル構成:
1. 従業員マスタ
2. 資格マスタ
3. 資格記録

### 7.2 Lark Baseで十分な領域

| 資格管理 | ◎ | 日付トリガーで自動通知 |

## 8. 最終推奨

### 8.1 現時点での推奨

**Lark Base を継続使用**

理由:
1. 既に「シンプル建設業務管理パッケージ v2.0」が構築済み
2. 資格管理・力量管理はLark Baseが最適
3. 中小規模の工程管理には十分
4. 追加コスト・学習コストなし

### 8.2 将来的な拡張案

```
Phase 1 (現在): Lark Base 単独
- 現行パッケージの運用定着
- オートメーション活用
- 運用ノウハウ蓄積

Phase 2 (半年後): 評価期間
- Meegle無料版で試験導入
- 大規模案件で試用
- 効果測定

Phase 3 (1年後): 判断
- 効果があればMeegle本格導入
- ハイブリッド運用へ移行
```

### 8.3 Meegle導入の判断基準

以下の条件が3つ以上該当する場合、Meegle導入を推奨:

- [ ] 同時進行プロジェクトが常時5件以上
- [ ] 1プロジェクトの工程数が50以上
- [ ] 工程間の依存関係が複雑
- [ ] リソース（人員・機材）の最適配置が課題
- [ ] 工程遅延による損失が大きい
- [ ] 複数部門・協力会社の協働が多い
